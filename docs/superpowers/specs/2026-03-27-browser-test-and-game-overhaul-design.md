# 浏览器全量测试 + 玩法深改 + 门户/PWA升级 设计文档

## 背景与目标
- 目标：在 Chrome 最新版 PC 端完成全量浏览器测试，修复问题，并分阶段完成玩法深改与门户/PWA升级。
- 体验方向：更爽、更有策略、更随机。
- 数据策略：允许不兼容旧存档，直接清空旧存档。

## 版本与环境基准
- 浏览器：以 2026-03-27 当日 Chrome Stable 版本为验收基准。
- 平台：仅 Windows 10/11 + Chrome Stable（移动端与其他浏览器不在验收范围）。

## 范围
- 页面：`index.html`、`games/*.html`、`offline.html`、`sw.js`、`manifest.json`。
- 玩法：所有现有游戏（`cultivation`/`lifesim`/`cardtower`/`cardbattle`/`cardcollect`/`knife`/`guigu`）。
- 环境：Chrome 最新版（PC），全新用户数据（清空 localStorage/缓存/SW）。

## 非目标
- 不引入第三方运行时依赖。
- 不新增“完全独立的新游戏”作为阶段1-3的目标（仅在后续讨论）。

## 分阶段计划（方案C）
- 阶段1：全站浏览器测试 + P0/P1 修复 + 统一基础体验。
- 阶段2：选择 3 个游戏进行深改（爽感/策略/随机）。
- 阶段3：剩余游戏逐个深改。
- 阶段4：门户体验升级 + PWA/离线优化。

## 阶段1：测试与修复流程
1. 环境清理：清空 localStorage/缓存/SW。
2. 首页与导航：确认首页加载、动画/设置、排行榜切换、卡片进入各游戏页。
3. 玩法主循环：进入玩法 -> 新开局 -> 完成一次核心循环 -> 保存/退出/重进验证。
4. 关键交互：设置面板、按钮可用性、数值显示、存档/读档、排行榜更新。
5. 控制台错误：作为硬指标必须清零。

## 玩法最小核心循环定义（验收勾选）
- cultivation：创建/进入存档 -> 进入秘境战斗 -> 完成一场胜负 -> 结算奖励 -> 返回主界面。
- lifesim：新开人生 -> 执行至少 3 次行动 -> 触发至少 1 次事件/结算提示 -> 进入下一岁。
- cardtower：进入地图 -> 进行一场战斗 -> 结算奖励 -> 选择下一节点并进入。
- cardbattle：开始对战 -> 至少 5 回合出牌/攻击 -> 结束胜负 -> 返回大厅。
- cardcollect：进入章节战斗 -> 完成一场胜负 -> 结算掉落/奖励 -> 返回章节界面。
- knife：进入游戏 -> 完成至少 1 个波次 -> 结算/过场 -> 继续或死亡。
- guigu：创建角色 -> 完成一次修炼或探索 -> 时间推进与日志记录 -> 返回主界面。

## P0/P1 分级标准
- P0：无法进入页面/玩法、卡死无法继续、功能按钮不可用导致无法推进。
- P1：核心逻辑错误（数值/存档/结算/排行榜明显异常）。
- P2：体验/布局/性能与文案问题（不影响继续推进）。

## 记录方式（交付口径）
- 仅对外输出“问题 + 修复摘要”。
- 内部最小记录模板（用于验证，不对外展示）：
  - 页面/游戏
  - 复现步骤
  - 期望/实际
  - 控制台错误
  - 修复摘要

## 修复策略
- P0（阻断）与 P1（逻辑错误）优先。
- 最小可行修复，优先修改现有文件。
- 不隐藏错误，失败直接暴露日志，便于排查。

## 阶段2/3：玩法深改标准
- 核心循环重写：减少无效等待，反馈更快。
- 策略提升：增加明确可控的选择点（构筑/资源分配/风险收益）。
- 随机提升：引入可预期但不可控的事件/掉落/对手变化。
- UI/反馈升级：数值与结算清晰，避免猜测。
- 存档：阶段2起清空旧档，直接进入新版体验。

## 玩法深改可验证指标（最低基线）
- 至少新增 2 个可选策略分支（构筑/资源分配/风险收益）。
- 随机事件有明确触发条件与提示（非静默变化）。
- 单局平均无操作等待不超过 10 秒（可由战斗/回合驱动验证）。

## 阶段4：门户与PWA升级方向
- 门户：统一设置中心（动画/字体/难度偏好/音效）与跨游戏同步。
- 首页：卡片信息更清晰（进度/联动提示/最近游玩）。
- 排行榜：展示稳定性与可读性优化。
- PWA：明确缓存策略，版本升级提示，离线可用性稳定。
  - 约束：修改 `sw.js` 时必须 bump `CACHE_VERSION`，并同步更新 `STATIC_ASSETS`。

## 验收标准
- Chrome 最新版 PC 端无 P0/P1。
- 控制台无未处理错误（`console.error/uncaught` 为硬指标；`console.warn` 允许但需列入问题清单）。
- 所有游戏可进入并完成一次核心循环。
- 阶段1：离线模式可打开首页与已缓存游戏页（基础可用性）。
- 阶段4：缓存策略与升级提示完善。

## 数据清档策略
- 触发时机：版本升级且检测到存档 schema 变更时清档。
- 用户提示：进入前提示一次并要求确认，避免无声清档。

## 阶段2选择依据
- 依据当前问题密度、玩法可扩展性、改动成本与可复用机制优先。

## 性能基线
- 首屏交互可用不超过 2 秒。
- 关键按钮点击后 200ms 内有反馈（动效/提示/状态变化）。
